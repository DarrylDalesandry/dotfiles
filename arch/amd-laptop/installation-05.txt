At this step, you may also have GNOME, KDE, or Cosmic installed. I'm still
running from the TTY, so open a terminal and follow along.

Sound is not going to be that difficult to manage. This mostly involves
installing a few packages that automatically connect to each other. These
cores applications are:

ALSA: Advanced Linux Sound Architecture
-- ALSA is responsible for detecting that speakers and microphones exist

Pipewire: An audio framework for both sound and video
--- Pipewire directs sound from apps to the speakers that ALSA detects

Wireplumber: A session and policy manager for Pipwire
--- Wireplumber allows control of what speakers sound is sent to, and allows
    changing the volume of the speaker, including mute, among other things


To get these all setup is pretty simple. Way back on the first page, the
package "alsa-utils" was already installed, so the audio drivers already
exist.

We then need to install all needed pipewire apps:

------------------------------
# sudo pacman -S pipewire
# sudo pacman -S pipewire-alsa
# sudo pacman -S pipewire-pulse
# sudo pacman -S pipewire-jack
# sudo pacman -S wireplumber
# sudo pacman -S rtkit
# sudo systemctl enable --now rtkit-daemon
------------------------------

Before pipewire, the audio application that managed sound was pulseaudio. Some
applications still use pulseaudio, so the "pipewire-pulse" package allows
sound intened for pulseaudio to be played through pipewire. The same is true
for JACK, which is intended for MIDI and professional sound that some desktop
applications use. The package "pipewire-jack" converts audio intended for JACK
to be used in pipewire.

Wireplumber provides access to the command "wpctl". This is the terminal
command that allows volume to change, and sources to change. So if you are
listening to sound on your laptop speakers, and you want to change it to some
headphones, that is wireplumber that is responsible for this.

Because audio, and video as Pipewire does support it, needs to be done in
real time, the RealTime Kit needs installed too (rtkit). This allows real-time
scheduling for audio and video processes without needing root access. Think
of this as a way of allow real-time audio, like YouTube videos, to have
permission to jump ahead of other processes to play sound when requested, then
allow other apps to use the CPU after audio tools, like Wireplumber, are done
with it.

For more information on Wireplubmer, see the documentation at:
https://wiki.archlinux.org/title/WirePlumber

For specific control over audio using "wpctl", see the documentation at:
https://wiki.archlinux.org/title/WirePlumber#Keyboard_volume_control

A reboot is needed, and audio should be working.

To test, even if you boot into a tty, this can be verified by running:

------------------------------
# speaker-test -c 2 -r 48000 -t wav
------------------------------

This will play an audio test of the front left and right speakers, until it is
stopped with CTRL+C.

The command "speaker-test" is a part of "alsa-utils" that was installed earlier.




From: https://wiki.archlinux.org/title/Fprint

The Framework laptop has a fingerprint reader built into it. This can be enabled
if you want. The main advantage of registering a fingerprint reader, is that
if you type a "sudo" command, then you get the option to elevate by resting
your finger on the power button of the Framework laptop, which doubles as a
fingerprint reader. You then no longer need to type in a password in order
to provide elevation to "sudo" commands.

If your laptop is on a dock, with its lid closed, then you can skip the
fingerprint authentication by pressing CTRL+C, and then you'll be prompted to
enter a password. This is what will be setup next.

Fingerprints are the most useful if you know you will be using your laptop
out and about, where the fingerprint reader would be commonly used. But it is
not useful if you are docked and the laptop is out of reach, or the lid is
closed.

This is up to you if you want to install this or not. You don't have to.

------------------------------
# sudo pacman -S fprintd
# sudo pacman -S imagemagick
------------------------------

Double check that "libfprint" is an installed dependency, this is needed to get
fingerprint authentication to work.


From: https://wiki.archlinux.org/title/Polkit#Authentication_agents

The packages install the fingerprint reading app, a policy kit (polkit), and an
authentication agent for TTY sessions named "pkttyagent". These are built on
the toolkit Pluggable Authencation Modules (PAM). Graphical desktop environments
need a graphical authentication agent installed as well. If you are using GNOME
or KDE, then you already have a graphical agent installed.

COSMIC, on the other hand, does not contain a graphical authentication agent,
as the company that develops COSMIC, System76, does not prefer to use one. A
workaround for COSMIC is to install "lxqt-polkit-agent". This is not designed
for COSMIC, so it will look very out of place.


From: https://wiki.archlinux.org/title/PAM

PAM = allows the physical fingerprint reader to be an authentication method
polkit = allows fingerprints to be used on the OS as an authentication method

The fingerprint scanner captures "images" of a fingerprint to save locally on
the laptop. The "imagemagick" app is recommended in the Arch Wiki to be
installed, which may be needed for saving the fingerprints read as image files.

The location that fingerprints are saved, is at "/var/lib/fprintd". Only
elevated accounts can view this, which means user accounts cannot, so the "sudo"
command would be needed to view the contents.

PAM is first used to determine what actions permit a fingerprint to be scanned. 
Fprintd is run, to capture an image of a fingerprint as requested by PAM. After
capturing the fingerprint image, fprintd then compares it to the fingerprint
images on device to determine if the fingerprint is genuine or not, followed
by polkit permitting elevated access if confirmed to be a match.

First, we need to tell Arch what we are permitting fingerprint authentication
to be used for. This will be using PAM.

------------------------------
# cd /etc/pam.d
# sudo nvim ./sudo
------------------------------

In the "/etc/pamd.d/sudo" file, there will be a list of items already listed.
At the top, above the "auth" text, type:

------------------------------
auth        sufficient      pam_fprintd.so
auth        sufficient      pam_unix.so try_first_pass nullok
------------------------------


The file should look similar to:
------------------------------
auth        sufficient      pam_fprintd.so
auth        sufficient      pam_unix.so try_first_pass nullok
auth        include         system-auth
account     include         system-auth
session     include         system-auth
------------------------------

Save and quit. This will then allow fingerprints to be used to run the "sudo"
command.

If that fails, then the fallback is "pam_unix.so" which asks for a password.
The "try_first_pass" means that this authorization entry will only try when
the fingerprint fails, which "nullok" means "null is OK", or that empty entries,
like pressing CTRL+C, will fallback to the second rule.

The default rule for "polkit" is in the file:
  "/usr/share/polkit-1/rules.d/50-default.rules".

This location is not accessible by a user account. But, it returns that the
password expected is the password for the members of the "wheel" group. When
the user account was created back in step 2 of this guide, we already added
the user account to the "wheel" group.


Next, is to then get the user account enrolled with their fingerprints.

------------------------------
# sudo fprintd-enroll {username}
------------------------------

Where {username} is your account's username. You could use $(whoami) in the
place of {username} if not remembered.

This will start the process of allowing fingerprint scanning to be run.

After completing, this can be tested by running the command:

------------------------------
# fprintd-verify
------------------------------

A finger will asked to be scanned, and then prove that it was verified.

If you reboot your computer, then your terminal should ask for a fingerprint
to be scanned if you run a "sudo" command.

All files in "/etc/pam.d" can be set similarly to allow fingerprints to be
permitted to do different tasks by the name of the file.

For example, if opening "/etc/pam.d/greetd", and making the first two lines:

------------------------------
auth        sufficient      pam_fprintd.so max-tries=1 timeout=30
auth        required        pam_unix.so try_first_pass nullok
------------------------------

Then this will show the display manager asking for a fingerprint to login.
If no fingerprint is provided in 30 seconds, it then asks for a password. Or,
if a fingerprint is not recognized, it fails over to asking for a password.

A fingerprint is OK to use to login, but if that fails, then a password is
absolutely required and there is no failing over.




Here we are, at Secure Boot. 

So, what is Secure Boot? This is a security standard that has a computer's UEFI
verify that images and files responsible for booting an operating system are
securely signed, using a signing key generated by the computer.

Microsoft is the company that created the Secure Boot standard, and they utilize
a Certificate Authority to allow Windows to boot on Secure Boot computers.
Without a Certificate from Microsoft, Windows may not load at all, and ask for
a Bitlocker decryption key to boot into Windows.

On Linux, the signed files are the Linux kernel, "vmlinuz-linux", and a boot
loader, "BOOTX64.EFI". The signing keys are enrolled into the UEFI of a
computer, and allow these trusted boot files to be used. Nowhere in here,
is a TPM used. TPM is not required in order for Secure Boot to work.

A TPM is a microcontroller used to verify that every piece of software and
firmware that is managed in an operating system's boot process has not been
tampered, by storing hashes of the kernel, and boot loader, on the TPM itself.
When all components are verified, then the TPM can decrypt an SSD with the Arch
Linux installation on it and allow it to be run on the computer.

Using a TPM, a dedicated hardware chip on a computer's motherboard or embedded
in a CPU, is considered more secure to store these hashes and keys on, than
not using a TPM. If not using a TPM, the hashes and files used for boot
integrity are stored on the SSD, which is considered an easier place to
maliciously retrieve these hashes and keys from than a TPM.

For more reading on TPM:
https://medium.com/before-you-launch/do-i-need-a-tpm-chip-device-attestation-secure-boot-b98a9e8a7db0

The "problem" that is solved by using Secure Boot, is to prevent an "evil maid"
attack.

I am not kidding: https://en.wikipedia.org/wiki/Evil_maid_attack

An Evil Maid attack involves someone having physical access to your computer,
where they compromise it with malicious files, firmware or kernels while the
operating system boots. I can understand that you may feel that you are never
in a position that would even make you susceptable to an Evil Maid attack, so
you may not think that you need secure boot.

But this next part is important:

you are installing Arch, and you want to wipe away all certificates stored on

Certain models of computers can become expensive bricks.

Be careful, look at Arch Linux's Wiki pages, and determine for yourself if

Let's start by making sure that the computer being used supports TPM, by running
the command:

------------------------------
# systemd-analyze has-tpm2
------------------------------

If the computer supports TPM2, then the first line will say "yes".

First, reboot your computer and acces your UEFI. Secure Boot will need to
be reset, or cleared, depending on how your computer's UEFI is made. Then,
Secure Boot will need to be disabled, or not enforced, depending on what is
written.

Here we go, let's get some keys created, sign some boot files and firmware, and
enroll the keys. The tool that I'll be using is "sbctl", a control app for
Secure Boot.

------------------------------
# sudo pacman -S sbctl
------------------------------

Then, the status of secure boot can be verified with:

------------------------------
# sbctl status
------------------------------

Most stuff will be showing big red X symbols. Clearing your TPM should make
the "Setup Mode" be set to "Enabled". Despite the big scary red X, we want
the TPM to be in "Setup Mode".

We also want "sbctl" to show itself as not being installed. That is OK for now.

------------------------------
# sudo sbctl create-keys --pk-keytype tpm --kek-keytype tpm --db-keytype file
# sudo sbctl enroll-keys -m -f
# sbctl status
------------------------------

The first command "create-keys" will make "owner" keys that will be used for
signing. Two of them will be stored on the TPM, but the "database" key, that
will remain on the computer to allow faster operations of signing. Both
Arch's Wiki, and the documentation of "sbctl" recommend for "db" to be a file.

PM = Platform key
KEK = Key Exchange Key
DB = Signature Database

The command will make all three.

The second command "enroll-keys -m -f" will enroll those keys, with the "-m"
including Microsoft certificates, and "-f" including the OEM firmware, in my
case, from Framework.

Having Microsoft keys enrolled alongside the generated keys makes sure that
hardware and drivers are safe, and won't be broken if they do not detect that
Microsoft keys are not present.

The third command "sbctl status" should now show that:
- sbctl is installed
- Setup Mode is now set to disabled, this is a good thing
- Secure boot is still disabled
- Vendor keys are from microsoft, builtin-db and builtin-kek

Keys can be verified as being a tpm type, or a file type, with the command:

------------------------------
# sudo sbctl setup --print-config
------------------------------

Next, our generated keys need to sign some firmware and kernel files:

------------------------------
# sudo sbctl sign -s /efi/vmlinuz-linux
# sudo sbctl sign -s /efi/EFI/BOOT/BOOTX64.EFI
------------------------------

With systemd-boot being used, and wanting systemd-boot to be able to update
itself, another file will need signed. The bootloader is being signed directly
so that if an update occurs, then files in "/efi" will be signed automatically.

------------------------------
# sudo systemctl enable systemd-boot-update
# sudo sbctl sign -s -o /usr/lib/systemd/boot/efi/systemd-bootx64.efi.signed /usr/lib/systemd/boot/efi/systemd-boot.x64.efi
# sudo bootctl update
------------------------------

What is also nice about using "sbctl", is that it adds a hook into our wonderful
"mkinitcpio -P" command to sign our boot files directly if new files are made.

Run the command:

------------------------------
# sudo mkinitcpio -P
------------------------------

And you'll see the output of the command say that the files are already signed.

Then, to make sure that all files are signed that need to be, run:

------------------------------
# sudo sbctl verify
------------------------------

Green checkmarks should be listed all the way down. This should now be safe to
reboot your computer, and enable Secure Boot.

After the computer boots back into Arch, then run the terminal command:

------------------------------
# sudo sbctl status
------------------------------

Green checkmarks should be seen all the way down.



That is all for this file. By now, you should have a decent Arch system up and
running, that is built by hand.
